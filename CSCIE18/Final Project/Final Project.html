    Final Project

    • Due Dec 15, 2015 by 12:15pm
    • Points 300

    Final Project

    Update: Because of issues with the W3C Mobile OK Checker, I have removed that specific metric from the rubric for the mobile site. Your mobile site is still expected to be mobile friendly, but we will not be using the W3C Mobile OK Checker. If you already have used the W3C Mobile OK Checker, please cite that in your report. Otherwise, please cite some principles you followed to make your mobile site mobile-friendly! (see Google's What makes a good mobile site?)

    Overview

    Create a course catalog Web site for the Harvard University Faculty of Arts and Sciences. The site should contain a full-featured XHTML site, plus content in Mobile, and PDF formats. The courses should be grouped in a variety of ways: by department, by meeting day, by meeting time, and by instructor. Your site should have a navigation structure that lets the user easily move about the site. You will also need to submit a brief report explaining the features of your site and the technical means by which you achieved these results. You should eXist or other XML-based application (e.g. BaseX) to complete your final project.

    You may work in groups of 3 to 4 people. You may "self-organize" into groups. You may also work on the project individually as well.

    Group Roles

    • Project Manager, Organizer
    • Architect
    • UI and design for site, mobile site, PDF
    • XSLT and XPath experts
    • Writer

    Group Project Development Logistics

    • We will create additional Git repositories for each group (each group member will be able to push to the repository)

    Requirements Overview

    Your project will be evaluated in four areas.

    1. Report and Optional Presentation, 25%
    2. Sites and Content, 35%
      • HTML site
        • Browse and Search
        • Structure and Design
      • Mobile site
      • PDF content
    3. XML Technology Implementation, 30%
    4. Extraordinary Distinction, 10%

    Report and Optional Presentation

    The report will be critical to guiding us through your project. Your presentation and report should cover each of the major evaluation areas. Your report should explain how (and why) you have structured your project, and also indicate any features or solutions of interest. This should describe the approach, design, and implementation of the project. The report is a guide for us to your project -- if you want us to know about it, put it in the report! Reports should be in HTML or PDF.

    In addition, your report should reflect on the parts of your project you feel are very well done (and why), what parts are less well-done (and why), and a "lessons learned" section.

    Guidelines

    1. Report is a useful overview to the project and would be helpful to someone who would need to deploy, maintain, and extend the project. Report includes "what we're proud of" and "lessons learned" section.
    2. Report is moderately useful to guide the reader through the site and understand the technical and non-technical aspects of the project.
    3. Report is only marginally useful to guide the reader through the technical and non-technical aspects of the project

    Sites and Content

    HTML Site

    From the XML source files, use XML technologies to create a full-featured Web site. The HTML pages should be valid (HTML5 or XHTML 1.0 Transitional or XHTML 1.0 Strict) -- if not, let us know why! CSS should be used for formatting and styling. 

    Searching and Browsing

    Search fields should be presented so the user can easily find the types of courses they are looking for (see below). The user should have options to sort and/or group the search results by different criteria. 

    The course lists should be organized in several different ways, with each presenting a linked list of courses in summary form. The hypertext link should go to a course "detail" page that lists complete information about the course. First-level browsing of courses should be possible:

    • by Department and Course Group
    • by Day and by Time. This organization is useful if a person is looking for courses at a particular day, time slot, or day/time slot.

    You should handle the case where many results are returned and need to be "paged"

    Rubric

    1. Sufficient browse/search fields and groupings are implemented.
      • Navigation is clear on each page and driven dynamically.
      • The following types of questions can be easily answered:
        • What are all the courses offered in a given department?
        • What are the courses offered in a given department for a given term?
        • What are all the French courses offered in the Romance Languages and Literature Department?
        • I have an open slot on MWF at 10am in the fall term -- what courses could fill that slot?
        • What are all the courses that Robert Lue teaches?
        • I'm looking for a course about "leadership" -- what are the options?
    2. Most of the groupings above are present and most of the questions can be answered. Navigation is not detailed.
    3. Few of the groupings are present and the questions are difficult or cannot be answered. Navigation is weak.

    Web site guidelines:

    1. Site is a quality "beta" or better. Pages validate. Titles, keywords, descriptions are all ready for search services. Navigation provides the user information about where they are in the site and where they can go.
    2. Site is in a low quality "beta" state. It is ready for people to look at and understand how the site would work in production. There are still details to work out, but a production launch is in the very near future.
    3. Site is in a "prototype" state. Many details are left to implement. It is ready for the developers to show to others, but not for others to "drive themselves".

    Mobile Site

    To accommodate users of handheld devices, create a site that reflects the organization and grouping of the XHTML site above, except the pages should be appropriate for a mobile device. Also, since this is a mobile site, you may wish to have simpler page structure, fewer images, and simpler navigation. These pages should be designed with small devices in mind, both in terms of screen width and page 'length'. Adopting a responsive CSS framework may very well solve your mobile site issue!

    Rubric

    1. Content is appropriate for mobile device in terms of display width and content length. Navigation is appropriate for a mobile device. It is possible to browse through courses for a given department and to see detailed information about a specific course that is selected. W3C mobileOK Checker is at or above 70%
      Updated: At least two principles of making a good mobile site were cited and followed.
    2. Content and navigation is mostly appropriate for a mobile device (mobileOK checker at least 50%).
      Updated: One principle of making a good mobile site was cited and followed.
    3. Content and navigation is inappropriate for a mobile device (mobileOK checker less than 50%).
      Updated: No principles of making a good mobile site were cited or followed.

    PDF

    There should be a PDF document available that contains detailed information about the courses of any course listing in the HTML site.

    Rubric

    1. Each page in the HTML site has a PDF equivalent. Long PDF documents have a hyperlinked table of contents. Sections are used to logically group courses together. Detailed course information is provided.
    2. PDF availability is limited to a few 'views' such as by department, course group, or individual course. Long PDF documents have a hyperlinked table of contents. Sections are used to logically group courses together.
    3. PDF availability is limited. Courses tend to run together and a table of contents is not provided. Detailed information about courses are not provided.

    XML Technology Implementation

    Here we are expecting appropriate use and applications of XML technology to solve this problems of the project in a smart and flexible way.

    Are data and presentational elements appropriately separated?
    Are your structures flexible and data-driven (instead of hard-coded)?
    How do you make use of xsl:import and/or xsl:include to reduce repetition and gain consistency?
    Do you make use of XQuery functions to keep things DRY?
    Are choices between XQuery and XSLT consistent and clear?
    What if new departments were created - would this require a code change?
    How easy would it be to accommodate additional data fields about courses (e.g. if requirements fulfillment were provided as part of the course feed)?
    How easy would this be to turn into a University Catalog if you received data about courses from other schools within Harvard (Law, Business, Extension, Education, Design, Medical, Public Health, Government, etc.)
    How easy would it be to add a new output format (for example, ODF, Excel, eBook)?

    Rubric

    1. Project is well-structured and would be straight-forward to extend. "DRY" (Don't Repeat Yourself) principles are followed. Someone tasked with extending the project would be delighted at how easy it was to add features and functionality.
    2. Project has been structured adequately. "DRY" has been followed mostly. Someone extending the project would be able to do so without extraordinary efforts.
    3. Project is not structured well. "DRY" has been seldom followed. Someone extending the project would have large amounts of work to do.

    Extraordinary Distinction

    Highlight any particular features, solutions, or implementations that you feel deserve recognition for extraordinary distinction. What useful and creative features does your project include that go beyond the basic requirements? Are there creative technology solutions behind your project?
